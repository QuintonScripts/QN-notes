* Encapsulation is one of the four pillars of OOP.
* The four pillars are a common interview question.
* Java is **statically typed**:

  * Variables can only store values of their declared type.
* Java primitive types:

  * `byte`
  * `short`
  * `int`
  * `long`
  * `float`
  * `double`
  * `boolean`
  * `char`
* Wrapper classes (reference types) exist for each primitive:

  * `Integer`, `Double`, `Character`, etc.
* Wrapper classes provide useful helper methods.
* Converting primitives to wrapper objects is **boxing**.
* Converting wrapper objects back to primitives is **unboxing**.
* Java performs boxing/unboxing automatically.
* Objects (reference types):

  * Store data (state).
  * Provide methods to operate on that data.
* Object **state** refers to the values currently stored inside the object.
* Good software controls and protects object state.
* Fix compiler/linter errors immediately.
* Letting errors accumulate makes debugging much harder.
* Run your code frequently while developing.
* Begin writing unit tests early.
* A **function** is reusable code.
* A **method** is a function that belongs to an object.
* Java accesses methods using **dot notation** (`object.method()`).
* An object is:

  * An instance of a class.
  * A collection of data and methods.
* Encapsulation means:

  * Restrict direct access to object data.
  * Bundle data with the methods that operate on it.
  * Allow objects to control their own state.
* External code should **not** directly modify an object's internal data.
* Access modifiers control visibility:

  * `public` → accessible everywhere.
  * `protected` → class, package, subclasses.
  * package-private (default) → class and package only.
  * `private` → class only.
* `private` fields are commonly accessed through **getters** and **setters**.
* Getters/setters protect object state instead of exposing variables directly.
